Kapitel 01

Über die App

Talentleihe Berlin ist eine Web-App, die Verbundausbildung im Berliner Handwerk organisiert und vermittelt. Betriebe können Azubis vorübergehend an andere Betriebe ausleihen (Einsätze), und Azubis können sich aktiv mit eigenen Talent-Angeboten vorstellen. So entstehen flexible Lernarrangements, die über den eigenen Ausbildungsbetrieb hinausgehen.

Die App ist im Rahmen des HSBA-Projekts „Talent trifft Handwerk" entstanden.

Zielgruppe

RolleBeschreibung
Talent (Azubi)Auszubildende im Handwerk, die sich für befristete Einsätze bei anderen Betrieben anbieten oder auf Einsatzangebote bewerben
BetriebHandwerksbetriebe, die Einsätze für Azubis ausschreiben oder Talent-Angebote von Azubis suchen

Zweck: Verbundausbildung

Verbundausbildung bedeutet, dass mehrere Betriebe gemeinsam ausbilden. Durch Talentleihe können Azubis Fertigkeiten erwerben, die der eigene Ausbildungsbetrieb nicht abdeckt.

Regionalbeschränkung: Die App beschränkt sich bewusst auf Berlin und Ostbrandenburg – Zuständigkeitsgebiet der Handwerkskammer Berlin und der Handwerkskammer Frankfurt (Oder), Region Ostbrandenburg.

Kapitel 03

Nutzerrollen

Die App kennt zwei Profiltypen. Nach der Registrierung legt jeder Nutzer einmalig ein Profil an und wählt dabei seine Rolle. Die Rolle bestimmt, was in der App sichtbar ist und was getan werden kann.

Talent · Azubi

Was Talents sehen

Auf der Talentleihe-Seite ausschließlich Einsatz-Angebote von Betrieben (Typ einsatz) – als Liste oder Karte.

Aktionen

  • Profil anlegen und bearbeiten
  • Bewerbungsunterlagen hochladen
  • Eigene Talent-Angebote erstellen
  • Auf Einsatz-Anzeigen bewerben
  • Bewerbungen zurückziehen
  • Abgeschlossene Einsätze bewerten

Bewertungskategorien

Zuverlässigkeit · Arbeitsumfeld · Wissensvermittlung

Betrieb

Was Betriebe sehen

Auf der Talentleihe-Seite ausschließlich Talent-Angebote von Azubis (Typ talent_angebot) – nur Listenansicht.

Aktionen

  • Betriebsprofil anlegen und bearbeiten
  • Einsatz-Anzeigen erstellen und löschen
  • Eingehende Bewerbungen annehmen/ablehnen
  • Bewerbungsübersicht verwalten
  • Abgeschlossene Einsätze bewerten

Bewertungskategorien

Zuverlässigkeit · Lernbereitschaft · Pünktlichkeit

Nutzer ohne Profil: Wer eingeloggt ist, aber noch kein Profil angelegt hat, sieht alle Anzeigen beider Typen und erhält den Hinweis, das Profil zu vervollständigen.

Registrierung & Login

  • Registrierung (/registrieren): E-Mail + Passwort; anschließend Versand einer Verifizierungs-E-Mail. Passwort-Toggle vorhanden.
  • Login (/login): Nach dem Login Weiterleitung auf /konto.
  • E-Mail-Verifizierung: Nutzer ohne bestätigte E-Mail sehen ein Banner mit Link zum erneuten Versenden.
  • Passwort vergessen: E-Mail-Adresse → Reset-Mail → Link führt zu /passwort-neu.

Konto (Profil & Hub)

Die Konto-Seite (/konto) ist der zentrale Hub nach dem Login.

  • Beim ersten Login: Formular zur Anlage des Profils (einmalige Rollenwahl: Talent oder Betrieb)
  • Schaltfläche „Profil bearbeiten" öffnet das Formular mit vorausgefüllten Werten
  • Hub-Karten zu Meine Anzeigen, Meine Bewerbungen, Eingegangene Anfragen
  • Bewerbungsunterlagen-Verwaltung (nur Talents)
  • Eigene Bewertungsübersicht als Sternkarten
  • Logout sowie Konto löschen (nach E-Mail-Bestätigung)

Kapitel 06

Filter & Suche

Die Talentleihe-Seite (/anzeigen) bietet einen ausklappbaren Filterbereich.

Gewerk-Filter

  • Mehrfachauswahl (Multi-Picker mit Suche)
  • Vorausfüllung aus Profil: Talent → eigenes Gewerk; Betrieb → alle Gewerke vorausgewählt
  • Matching via gewerkStamm()-Funktion (toleriert Singular/Plural-Varianten)

Ort / Umkreis-Filter

  • Freitextfeld: Stadt, Stadtteil oder PLZ
  • Umkreis: Schieberegler 0–200 km (Schritte à 5 km), Standard: 50 km
  • Geocoding via OpenStreetMap Nominatim API (mit „, Deutschland"-Ergänzung)
  • Distanzberechnung: Haversine-Formel (Luftlinie)
  • Debounce: 500 ms; Ergebnisse werden gecacht (im gleichen Seitenaufruf)

Datum-Filter

Zwei Felder: „Verfügbar ab" und „Verfügbar bis"

Überschneidungslogik: Anzeigen mit Enddatum: Einsatzzeitraum muss den Filterzeitraum überschneiden. Anzeigen ohne Enddatum: Startdatum muss im Filterzeitraum liegen. Das „bis"-Datum wird serverseitig gefiltert; restliche Bedingungen laufen clientseitig (Appwrite unterstützt kein OR).

Handwerkskammer-Filter

  • Dropdown: „Beide Kammern" oder eine der zwei Berliner Regionalkammern
  • Standardmäßig werden immer nur Anzeigen aus diesen zwei Kammern geladen

Mindestalter-Filter

  • Optionen: „egal", „nur ohne Mindestalter", oder konkrete Altersstufe
  • Serverseitiger Filter über Appwrite-Query

Kartenansicht

  • Nur für Talents verfügbar
  • Anzeigen auf interaktiver Karte (AnzeigenMap-Komponente)
  • Kammer-Gebiete als Flächen eingeblendet
  • Klick auf Pin → Anzeige-Detailseite
  • Koordinaten werden einmalig geocodiert und gecacht

Entwicklungsumgebung einrichten

Voraussetzungen

  • Node.js ab Version 18
  • npm (wird mit Node.js mitgeliefert)
  • Zugang zur Appwrite-Cloud-Instanz (oder lokale Appwrite-Instanz)
  1. Repository klonen
    bashgit clone <repository-url>
cd Talentleihe-Berlin
  2. Abhängigkeiten installieren
    bashnpm install
  3. Umgebungsvariablen anlegen

    Datei .env.local im Projektroot anlegen:

    envVITE_APPWRITE_ENDPOINT=https://cloud.appwrite.io/v1
VITE_APPWRITE_PROJECT_ID=<deine-project-id>
  4. Dev-Server starten
    bash# Empfohlen (alle Betriebssysteme):
npx vite

# Alternativ:
npm run dev

# Windows / PowerShell (falls npx nicht greift):
node_modules\.bin\vite.cmd

    Die App ist dann unter http://localhost:5173 erreichbar.

    Windows-Hinweis: Der direkte vite-Befehl schlägt unter PowerShell manchmal fehl. Die explizite Angabe node_modules\.bin\vite.cmd umgeht dieses Problem zuverlässig.
  5. Build erstellen (optional)
    bashnpm run build

    Der Build wird im Ordner dist/ abgelegt.

Kapitel 08

Deployment

Die App wird über Appwrite Sites gehostet und ist mit dem GitHub-Repository verbunden.

Ablauf

  1. Änderungen pushen
    bashgit push origin main
  2. Automatischer Build

    Appwrite Sites erkennt den Push via GitHub-Webhook und startet npm run build automatisch.

  3. Live-Schaltung

    Nach erfolgreichem Build ist die neue Version innerhalb von 1–3 Minuten live.

Build-Status prüfen

Im Appwrite Dashboard unter Sites → aktives Deployment auswählen. Dort sind Build-Logs, Status und Deployment-URL einsehbar.

Umgebungsvariablen im Deployment

Die Variablen VITE_APPWRITE_ENDPOINT und VITE_APPWRITE_PROJECT_ID müssen auch in den Appwrite-Sites-Einstellungen unter Environment Variables hinterlegt werden. Sie werden beim Build eingebettet (Vite import.meta.env).
Kapitel 09

Bekannte Besonderheiten & Hinweise

TopNav ist custom (fixed, zIndex 9999)

Die App verwendet keine native Ionic-Navigation, sondern eine vollständig eigene TopNav-Komponente.

  • Position: fixed, z-index: 9999
  • Der IonHeader auf Einzelseiten liegt hinter der TopNav und ist nicht sichtbar – er wird für Hilfszwecke beibehalten
  • Seiteninhalte haben oben ausreichend Abstand, damit sie nicht hinter der TopNav verschwinden

Zurück-Button: eigene ZurueckButton-Komponente

Der native Ionic IonBackButton wird nicht verwendet. Die eigene ZurueckButton-Komponente navigiert sowohl auf Mobilgeräten als auch im Browser korrekt im Routing-Stack.

Mehrere Gewerke für Betriebe: kommagetrennt im gewerk-Feld

Das gewerk-Feld der profiles-Collection ist ein einzelner String. Für Betriebe mit mehreren Gewerken wird er als kommagetrennte Liste genutzt:

ts// Lesen:
gewerk.split(",").map(s => s.trim()).filter(Boolean)

// Speichern:
selectedGewerke.join(", ")

Dokument-Downloads brauchen JWT (Cross-Origin Auth)

Download-URLs aus dem Appwrite Storage Bucket benötigen ein kurzlebiges JWT als Query-Parameter, um Cross-Origin-Authentifizierungsprobleme zu lösen.

  • Client erstellt kurz vor dem Download ein JWT
  • JWT wird als Query-Parameter an die Download-URL angehängt
  • Appwrite akzeptiert den Request auch Cross-Origin

Profil-Bug-Fix: saveProfile prüft Appwrite direkt

Problem: Bei schnellem Doppelklick auf „Profil speichern" könnte die App fälschlicherweise ein zweites Profil anlegen, weil der lokale Auth-State noch nicht aktualisiert ist.
Fix: saveProfile prüft immer zuerst direkt in Appwrite (nicht im lokalen State), ob bereits ein Profil für die User-ID existiert, bevor es ein neues anlegt.

Anzeigen warten auf profileLoading (Race Condition Fix)

Die Talentleihe-Seite würde bei sofortigem Laden kurz alle Anzeigen beider Typen anzeigen, bevor der Profil-State fertig ist. Der Fix:

typescript// Aus Anzeigen.tsx
if (profileLoading) return;

Anzeige-Typ ist unveränderlich nach dem Erstellen

Der type einer Anzeige (einsatz / talent_angebot) richtet sich nach dem Profiltyp des Erstellers und kann nachträglich nicht geändert werden.

Regionalbeschränkung auf Berlin und Ostbrandenburg

Alle Anzeigen-Abfragen enthalten einen festen serverseitigen Filter:

typescriptQuery.equal("handwerkskammer", [...BERLIN_REGION_KAMMERN])

Anzeigen anderer Kammern werden weder geladen noch können sie erstellt werden (das HWK-Dropdown im Formular enthält nur die zwei Berliner Regionalkammern).
